Executive Summary
Capacity ceiling: 10 concurrent users. At that tier the failure rate is 4.9% (just under the 5%
threshold) with an average response time of 1,617ms and throughput of 3.0 RPS. Beyond 10
users, response times degrade catastrophically while throughput stays flat: 30.6s avg at 100
users, 104.7s at 500 users, 147.5s at 1K users, all at the same 3.0-4.4 RPS. At 10K users the
system collapsed entirely with 99.4% failures. This is the first recorded load test run, so no
prior-week comparison is available.
The flat RPS (~3 requests/second) across all tiers from 10 to 1,000 concurrent users is the
defining signal. The server processes requests at a fixed rate regardless of how many clients
are waiting. This pattern points to a serialized bottleneck: either a database connection pool
capped at a very low number, a single-threaded request handler, or a global lock in the
middleware. Two endpoints were already slow before load was applied: GET /api/Audit (11.0s
at zero concurrency) and GET /api/Holiday (7.5s at zero concurrency). POST /api/Schedule
returned HTTP 500 before the test started, meaning it was broken independent of load. These
three endpoints likely amplified degradation under concurrency by holding connections longer
than others.
Recommendations: (1) Profile the DB connection pool size and increase it from the current
setting (likely 1-5) to at least 20-50 for the expected user base. (2) Add a database index on the
Audit table, the 11s response at zero load suggests a full table scan. (3) Investigate the Holiday
list endpoint for a similar missing index or unoptimized query. (4) Fix the POST /api/Schedule
500 error, which is a code-level bug unrelated to load. (5) Instrument the API with request-level
timing (middleware logging) to identify whether the bottleneck is in the application layer or the
database layer. Until the ~3 RPS throughput ceiling is resolved, scaling users beyond 10
concurrent will produce unacceptable response times.
